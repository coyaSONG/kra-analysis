<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client credentials -> auth dependency | Untrusted API keys enter the auth system and must resolve to one canonical caller identity. |
| auth dependency -> policy/accounting | Caller identity crosses into authorization and usage accounting, where type drift can create privilege or audit bugs. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-07 | E | `apps/api/dependencies/auth.py` public helpers | mitigate | Return `AuthenticatedPrincipal` from router-facing helpers so authorization and resource checks do not branch on mixed `APIKey`/principal types per D-07 and D-08. |
| T-01-08 | R | `apps/api/policy/accounting.py` reservation path | mitigate | Preserve the existing reservation/commit seam and principal-derived identifiers so usage events remain attributable after refactoring per D-09. |
| T-01-09 | D | API key backend failures | mitigate | Keep backend lookup failures mapped to HTTP 503 and preserve daily-limit checks so auth cleanup does not turn storage outages into silent allows. |
</threat_model>
